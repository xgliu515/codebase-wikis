# 单请求 trace 导览：总览

## 这次跟踪什么请求

我们盯住一个具体动作，从头到尾看它在 OpenClaw 内部走过的每一行代码：

> 用户在浏览器里打开 WebChat 页面，在输入框敲下「你好」并按发送。OpenClaw 用一个 Anthropic 模型生成一句简短回复，回复显示回 WebChat 界面。

整条链路结束。没有别的。

### 为什么挑这一条

OpenClaw 是个把二十余种消息渠道统一接到一个 LLM 助手的网关，功能面非常宽：工具调用、跨渠道转发、媒体生成、定时任务、插件市场……如果一上来就跟踪一个「带工具调用、跨三个渠道、附带图片」的请求，你会在三十个分支里迷路，学不到主干。

这条「你好」请求被选中，是因为它同时满足三个条件：

- **最简**。一条短文本进、一条短文本出。无工具调用、无附件、无跨渠道路由、无媒体。每一步都只有「主干」没有「岔路」，岔路留到第 7 段「分支与延伸」里点到为止。
- **真实**。这不是为了教学编造的玩具路径。任何人装好 OpenClaw、打开 WebChat 说句话，跑的就是这条线。
- **贯穿全栈**。它从 CLI 启动器（`openclaw.mjs`）一路穿过 gateway 控制面、WebSocket 连接层、RPC 方法注册表、入站消息分发器、会话系统、插件钩子、agent 运行时、LLM provider，最后回到 WebSocket 广播与会话持久化。走完这一条，你就摸清了 OpenClaw 的骨架。

把这条 trace 走完，你脑子里会有一张「请求在系统里流动」的地图。之后再看任何复杂功能，都是在这张地图上加岔路。

## 怎么读每一步：固定的 8 段模板

从 tour-01 到 tour-17，每一步都用同一套 8 段结构来讲。这不是排版洁癖——它强迫每一步都先把「问题」讲清楚，再讲「方案」，让你理解 OpenClaw 为什么这样设计，而不是只记住它长什么样。

| 段 | 标题 | 这一段回答什么 |
|----|------|----------------|
| 1 | 当前情境 | 上一步结束时系统处于什么状态？我们手上有哪些对象？ |
| 2 | 问题 | 这一步要解决的核心问题是什么？用一句话说清。 |
| 3 | 朴素思路 | 一个没读过这份代码的工程师，第一反应会怎么做？ |
| 4 | 为什么朴素思路会崩 | 朴素思路具体在什么场景下、以什么方式失败。不是泛泛说「慢」「不优雅」，而是讲清楚崩的机制。 |
| 5 | OpenClaw 的做法 | 在「问题」和「失败」都摆清楚之后，OpenClaw 实际怎么做。先建立张力，再给方案。 |
| 6 | 代码位置 | 真实的 `文件:行号` 引用，带一句话说明每处在干什么。 |
| 7 | 分支与延伸 | 这一步上挂着哪些我们这次没走的岔路？想深入的话去看哪一章参考文档？ |
| 8 | 走完这一步你脑子里应该多了什么 | 3-5 条具体收获。读完能不能复述出来，是这一步是否真的学到的检验。 |

读法建议：第 2、4、5 段是核心，第 6 段是去源码里对照的索引，第 8 段是自测清单。

## 17 步速览

| 步 | 标题 | 一句话 |
|----|------|--------|
| 01 | 敲下 `openclaw gateway` 之后 | `openclaw.mjs` 启动器做版本检查、compile cache、信号转发，再引导进 `src/entry.ts` 与 `run-main.ts` 命令路由。 |
| 02 | Gateway server 启动监听 | `startGatewayServer` 加载 `server.impl.ts`，启动 HTTP/WebSocket 监听器、构建 RPC 方法注册表、加载插件。 |
| 03 | WebChat 建立连接 | 浏览器与 gateway 完成 WebSocket 握手，`connect` 帧携带 token/password 完成连接级鉴权。 |
| 04 | 客户端发出 `chat.send` | WebChat 前端把「你好」封装成一个 `chat.send` RPC 请求帧发出去。 |
| 05 | RPC 方法注册表分发 | 消息处理器解析帧，按 `method` 名在注册表里查到 `chat.send` 的 handler 并调用。 |
| 06 | 构造 MsgContext | `chat.send` handler 把 RPC 参数翻译成一个标准的入站消息上下文对象。 |
| 07 | dispatchInboundMessage | 入站分发总协调器接管 MsgContext，统筹后续所有处理。 |
| 08 | 会话解析与加载 | 根据上下文定位到具体的 session，解出要用哪个 agent、哪个 model。 |
| 09 | message_received 钩子 | 插件在消息正式进入 agent 之前获得一次介入机会。 |
| 10 | 创建 ReplyDispatcher | 构造负责投递、重试、打字指示器的回复协调器。 |
| 11 | 进入 agent command | `runAgentCommand` 解析 agent 运行时，准备执行一轮对话。 |
| 12 | 构建 prompt 与上下文 | 把历史记录、系统提示、当前这条「你好」拼成发给模型的输入。 |
| 13 | 调用 LLM provider | 通过 provider 抽象向 Anthropic 发起流式推理请求。 |
| 14 | 流式事件发射与订阅 | 模型流式产出的 token 被包装成 agent 事件，被下游消费。 |
| 15 | 组装 ReplyPayload | 流式事件被累积、整理成一条最终的回复负载。 |
| 16 | 投递回 WebChat 并广播 | 回复经由 channel send 写回 WebChat，并通过事件广播给其他订阅者。 |
| 17 | 会话持久化 | 这一轮对话的转录文本写回 session 存储，下次对话能续上。 |

## 状态变量演进表

整条 trace 的本质，是一组关键状态对象逐步「出生、长大、被消费」。下表横向看是 17 步，纵向看是每个状态对象的生命周期。`-` 表示尚未出现，`✓` 表示已存在且未变，描述文字表示该步对它做了什么。

| 状态对象 | 01-02 | 03 | 04-05 | 06-07 | 08-09 | 10-11 | 12-14 | 15-16 | 17 |
|----------|-------|-----|-------|-------|-------|-------|-------|-------|-----|
| Gateway 监听状态 | 出生：HTTP/WS 监听器就绪 | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ |
| RPC 方法注册表 | 出生：注册表构建、handler 注册 | ✓ | 被查询：定位 `chat.send` | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ |
| WebSocket 连接 | - | 出生：握手 + 鉴权，连接级状态登记 | 承载 `chat.send` 帧 | ✓ | ✓ | ✓ | ✓ | 承载回复帧广播 | ✓ |
| MsgContext | - | - | - | 出生：RPC 参数翻译成入站上下文 | 被读取以解析会话 | 传入 agent | ✓ | ✓ | 提供转录元数据 |
| session | - | - | - | - | 出生：解析并加载，确定 agent/model | 被 agent 引用 | 提供历史 | ✓ | 被写回 |
| ReplyDispatcher | - | - | - | - | - | 出生：投递/重试/打字指示器协调器 | 接收 agent 事件 | 执行投递 | 触发持久化 |
| agent runtime | - | - | - | - | - | 出生：`runAgentCommand` 解析运行时 | 运行一轮对话 | 收尾 | - |
| LLM 流 | - | - | - | - | - | - | 出生：Anthropic 流式调用，事件逐个发射 | 流结束 | - |
| ReplyPayload | - | - | - | - | - | - | 累积中：事件不断追加 | 出生→定稿：组装成最终回复 | - |
| 持久化记录 | - | - | - | - | - | - | - | - | 出生：转录写回 session 存储 |

读这张表的方式：找一个状态对象，横着看它什么时候出生、被谁读、被谁改、最后归于何处。比如 `ReplyDispatcher` 在第 10 步出生，在第 14 步开始接收 agent 事件，在第 16 步真正把回复投递出去，并在第 17 步触发持久化——它的生命周期横跨了 trace 的后半段。

## 如何使用这份导览

- 按 01→17 顺序读。每一步都假设你已经读完了前一步，会直接从「当前情境」接上。
- 第 6 段的所有 `文件:行号` 引用都锁定在 commit `50a2481652`（tag `v2026.5.18`）。行号是真实的，可以直接对照源码。
- 想就某个子系统深挖时，跟着第 7 段的章节链接去读对应的参考章节——导览给的是「这一条请求怎么走」，参考章节给的是「这个子系统全貌」。
